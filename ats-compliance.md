# ATS Compliance Guide

## Overview

Most companies use Applicant Tracking Systems (Workday, Greenhouse, Lever, iCIMS, Taleo, BambooHR) to parse and filter resumes before a human ever sees them. A resume that looks great as a PDF can fail parsing entirely if it uses formatting that ATS parsers can't read.

This guide ensures every resume generated by this skill passes ATS parsing while still looking professional to human reviewers.

## DOCX Formatting Rules

### Layout

- **Single-column layout only.** No multi-column layouts, no side panels, no sidebars. ATS parsers read top-to-bottom, left-to-right. Columns cause content to be scrambled or lost.
- **No tables for layout.** Tables are for tabular data, not for arranging resume sections side by side. ATS parsers often skip table content or read it cell-by-cell out of order.
- **No text boxes.** Floating text boxes are invisible to most ATS parsers.
- **No images, logos, or graphics.** ATS cannot read images. This includes headshot photos, company logos, decorative lines made from images, and infographic-style skill ratings.
- **No headers or footers for critical info.** Many ATS parsers skip header/footer content entirely. Contact information, name, and all other essential content must be in the document body.

### Fonts

Use standard, widely available fonts that render consistently:
- **Recommended:** Calibri (11pt body, 12-14pt headers), Arial, Garamond, Georgia, Cambria
- **Avoid:** Decorative fonts, script fonts, custom fonts that may not be installed on reviewer's system
- **Never:** Fonts that encode text differently (some artistic fonts map characters non-standardly)

### Section Headers

Use exact, standard section headers that ATS systems expect. Don't get creative with section naming.

**Standard headers (use these):**
- Professional Summary (or Summary)
- Skills (or Core Competencies, or Technical Skills)
- Professional Experience (or Work Experience, or Experience)
- Education
- Certifications (if applicable)
- Projects (if applicable)
- Publications (if applicable)

**Avoid non-standard headers like:**
- "Where I've Been" instead of "Professional Experience"
- "What I Bring" instead of "Skills"
- "My Journey" instead of "Education"
- "The Toolbox" instead of "Technical Skills"

### Heading Levels

- Name: Heading 1 (or large bold text, 14-16pt)
- Section headers: Heading 2 (or bold, 12-14pt)
- Role titles: Heading 3 or bold body text
- Body text: Normal style, 10-11pt

Consistent heading hierarchy helps ATS identify sections correctly.

### Bullets

- Use standard bullet characters: `-` or `•` generated through proper list formatting
- Do NOT use unicode symbols (★, ▸, ►, ●) as bullet markers — some parsers misread them
- Do NOT use emoji as bullets
- Use proper Word numbering/bullet configuration, not manually typed characters
- Keep bullets to 1-2 lines. Longer bullets get truncated in some ATS summary views.

### Spacing and Margins

- Margins: 0.5" to 1.0" on all sides (0.7" is a good default)
- Line spacing: Single within sections, 6-12pt space between sections
- No excessive whitespace — ATS doesn't see it but human reviewers may see oddly sparse pages after parsing

### File Naming

```
FirstName_LastName_Resume.docx
```

- No spaces in filename (use underscores)
- No special characters
- Include "Resume" in the name so it's identifiable in bulk uploads
- DOCX format preferred over PDF for ATS parsing (unless job posting specifically requests PDF)

## Content Optimisation for ATS

### Keyword Matching

ATS systems score resumes partly on keyword density against the job description. Optimise honestly:

**Do:**
- Mirror exact JD terminology in the Skills section where the user genuinely has that skill
- Use the JD's preferred term (e.g., if JD says "Kubernetes" don't write "K8s" only — include both or use the full word)
- Include both abbreviations and spelled-out versions: "CI/CD (Continuous Integration/Continuous Deployment)"
- Place highest-priority keywords in the Skills section AND in experience bullets (some ATS weight skills sections differently from experience)

**Don't:**
- Keyword-stuff skills the user doesn't have
- Hide keywords in white text (this is detectable and will get the application rejected)
- Repeat the same keyword unnaturally many times
- Add a "Keywords" section — this looks spammy to human reviewers

### Skills Section Structure

Organise skills into 2-4 categories that match how the JD groups them:

```
Technical Skills: Python, SQL, Kubernetes, AWS, Terraform, CI/CD
Data & Analytics: Tableau, pandas, A/B testing, Statistical modeling
Leadership: Cross-functional coordination, Stakeholder management, OKR planning
Tools: Jira, Confluence, GitHub, Figma
```

- List skills as comma-separated items, not as a bulleted list (saves space, parses cleanly)
- Put the most JD-relevant skills first in each category
- Only include skills the user actually has and can discuss in an interview

### Dates Format

Use a consistent, parseable date format throughout:

- **Recommended:** "Jan 2022 - Present" or "January 2022 - Present"
- **Also acceptable:** "01/2022 - Present" or "2022 - Present" (year only)
- **Avoid:** Inconsistent formats (mixing "Jan 2022" with "2023-06")
- **Never:** Vague dates like "2022ish" or "Early 2023"

### Professional Summary

- Keep to 2-4 sentences
- Front-load with the most relevant qualifier for this specific role
- Include 2-3 high-priority keywords from the JD naturally
- Avoid generic filler: "results-driven professional with a passion for excellence"

### Contact Information

Place at top of document body (not in header/footer):

```
Full Name
City, State (no full address needed)
Phone | Email | LinkedIn URL (optional)
```

- Use standard separators (pipes `|` or line breaks)
- LinkedIn URL should be the custom URL if available (linkedin.com/in/name)
- Don't include age, photo, marital status, or other personal details

## ATS Pre-Submission Checklist

Run through this before finalising any resume:

```
[ ] Single-column layout, no tables or text boxes
[ ] Contact info in document body, not header/footer
[ ] Standard section headers used
[ ] Consistent heading hierarchy
[ ] Standard font (Calibri, Arial, Garamond)
[ ] Proper bullet formatting (not unicode symbols)
[ ] Keywords from JD present in both Skills section and experience bullets
[ ] Dates in consistent, parseable format
[ ] No images, logos, or graphics
[ ] File named FirstName_LastName_Resume.docx
[ ] All skills listed are truthful and user can discuss in interview
[ ] Professional summary is 2-4 sentences, not generic filler
```

## Format Priority

When generating resume outputs:

1. **DOCX** — Primary format. Best for ATS parsing and human editing. Always generate this.
2. **Markdown** — Internal working format and library storage. Always generate this.
3. **PDF** — Optional. Generate only if user requests. Some ATS parse PDF well, others don't. When generating PDF, ensure it's text-based (not a scanned image).

## Common ATS Failure Modes

Things that cause resumes to parse incorrectly or be scored at 0%:

- **Two-column layouts:** Content from right column gets appended after left column content, creating nonsensical text
- **Tables:** Section headers in one cell, content in another — parser may only see one
- **Text boxes:** Entire sections become invisible
- **Header/footer content:** Name and contact info vanish from parsed output
- **Image-based elements:** Skill bars, star ratings, progress circles — all invisible
- **Non-standard fonts:** Characters map incorrectly, creating garbled text
- **PDF from design tools (Canva, Figma):** Often export as image layers, not text — ATS sees a blank page
